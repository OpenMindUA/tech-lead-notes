# Definition of Outcome Done

**TL;DR:** **New commitment in SGE**, attached to the **Product**
(not the Increment). Describes **observable evidence measures** that
demonstrate **realized benefits** = "value validation". Pairs with
the renamed Definition of Output Done.

## Definition

> "Describes the observable evidence measures (quantitative or
> qualitative) required to demonstrate realized benefits, often
> referred to as value validation."

- Can be for the **overall Product** or for a **specific goal**.
- **Often best to define measures *before* realization starts** —
  avoids biases and mistaken interpretations.
- **Outcomes** confirm **user changes**.
- **Impact** confirms **organizational results**.

## Validation channels

- **Product telemetry** (Product can measure its own usage).
- **Strategic deployment** efficacy validation for the overall
  Product.
- A **combination** of both.

## Categories (favor direct evidence over circumstantial)

| Category | Examples |
|----------|----------|
| **Customer outcomes** | satisfaction trends, customer cost reduction, customer jobs addressed |
| **User outcomes** | task completion efficiency, engagement with new features |
| **Product Stakeholder outcomes** | customer / decision-maker / user metrics, time to release / learn / pivot |
| **Business impact** | compliance, cost reduction, market-share trends, organizational time to release / learn / pivot, satisfaction across all Products |
| **Scrum Team outcomes** | technical capability (psychological flow, release frequency, tooling, skill, debt, capacity), climate / culture |

## Authority

- The **Product Owner has the final say** on the Definition of
  Outcome Done.
- The Scrum Master helps the PO **express outcomes as measures**.

## UX / CX debt (related concept)

> "Sum of design and implementation choices that make a Product or
> service less usable, enjoyable, or effective."

Recognize, track, address. Increment quality alone doesn't capture
this — outcome measures do.

## When this applies

- Setting up a new Product's measurement plan
- Reviewing Product strategy at the Sprint Review
- Resolving "are we delivering value?" questions with evidence
- Addressing UX / CX debt with measurable signals

## Anti-patterns

- **No measures defined** — outcomes left to opinion
- Measures defined **after** realization starts — biased
- Only **circumstantial evidence** (e.g. number of features
  shipped) treated as outcome
- Conflating outcomes (user changes) with **outputs** (features
  shipped) or **impact** (org results)
- PO doesn't actually own the DoOD measures
- Increment ships Definition of Output Done while no outcome
  evidence is captured
