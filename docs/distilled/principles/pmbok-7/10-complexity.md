# Navigate complexity

**TL;DR:** Project complexity comes from three sources: people,
non-linear interactions, and ambiguity/uncertainty. There is no
distinct technique here — applying the other principles *is* how you
navigate complexity.

> The author considers this a redundant principle: each contributing
> source already has its own dedicated principle. PMBOK 7 keeps it
> as a separate principle for emphasis.

## Core ideas

- **Three sources of complexity** in projects:
    1. **People** — humans and their interactions create
       complexity (covered by Principles 1–3: stewardship,
       collaborative team, stakeholder engagement).
    2. **Non-linear interactions** — system effects that are not
       additive (covered by Principle 5: system interactions; also
       Principle 4: holistic value).
    3. **Ambiguity and uncertainty** — risks (covered by
       Principle 9: optimize risk responses).
- **No new techniques to introduce here.** Application of
  principles 1, 2, 3, 5, 9 *is* the practice of navigating
  complexity.

## When this applies

- When complexity is rising: don't look for a single new tool;
  re-apply the relevant primary principle (people / system / risk).
- When teaching the principles: use this as a synthesis frame.

## Anti-patterns

- Looking for a magical "complexity tool"
- Treating complexity as a separate workstream rather than a
  property of people, systems, and uncertainty
