# Tailor based on context

**TL;DR:** No universal PM system fits every project. Tailor a
chosen methodology to your size/culture/project — but tailoring is
system design, not cherry-picking.

## Core ideas

- **Match the system to the environment.** Copying another
  organization's mature system into a smaller/different company
  fails. The transplanted system was customized for a specific
  size, culture, history; reproducing it as-is breaks.
- **Tailor instead of rejecting.** "PRINCE2 is too expensive for us"
  usually means *un-tailored* PRINCE2. Properly tailored, even
  heavyweight methods can save cost.
- **Tailoring example.** PRINCE2 *checkpoint reports* are a
  *management product*, not a document format. In a small project
  this can be a phone call at fixed intervals. The decision about
  channel/depth is the act of tailoring.
- **Two-step tailoring (PMBOK 7).**
    1. **Organization-level** partial tailoring covering elements
       common to all projects.
    2. **Project-level** final tailoring for that project's
       specifics.
- **Cherry-picking ≠ tailoring.** Mixing fragments from different
  systems without internal consistency yields a "Frankenstein"
  system. Tailoring is holistic system design.
- **Tailoring is its own skill.** Most methods leave it to the PM,
  but few PMs are also system designers. DSDM addresses this with
  a dedicated **DSDM Coach** role.
- **When to tailor depends on the method:**
    - **Maximalist** (PRINCE2): tailor *upfront* — the system is too
      heavy to use untailored.
    - **Minimalist** (P3.express): use as-is, tailor *gradually* by
      enriching after experience.
- **Keep upfront tailoring minimum.** It's easy to design a fancy
  system that proves heavy in practice. Add elements as real needs
  surface.

## When this applies

- Organization rolling out a PM standard
- New project starting under an existing methodology
- A team complaining "the methodology doesn't fit"
- Choosing between a maximalist and minimalist starting point
- Defining a PM competency framework (include system-design skills)

## Anti-patterns

- Cherry-picking practices from multiple frameworks
- Using PRINCE2/PMBOK "by the book" without tailoring
- Heavy upfront tailoring → unused, ignored, or resented
- Assuming any senior PM can tailor a methodology
- "We're too small for X" instead of asking "how would X scale to us?"
