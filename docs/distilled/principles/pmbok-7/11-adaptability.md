# Embrace adaptability and resiliency

**TL;DR:** Predictive vs adaptive (Agile) approaches each fit a
different *kind of product*. PMBOK 7 is approach-agnostic for the
product, but the **PM system itself** must always be adaptive
because it deals with people.

## Core ideas

- **Predictive approach.** Design upfront, then build. Right when
  the output is fully knowable in advance and end-user feedback
  isn't a major source of design (Mars rover, bridge).
- **Adaptive (Agile) approach.** Build subsets, expose them to end
  users, collect feedback, decide next step. Right when output
  value depends on user acceptance (most software applications).
- **Both are valid.** Pick by output type, not by fashion. PMBOK 7
  avoids "waterfall" / "traditional" labels because they carry
  negative connotations.
- **Common myth.** PMBOK 7 has *not* "become Agile". It is
  *approach-agnostic* — supports both predictive and adaptive product
  development equally.
- **Two layers, not one.**
    1. **Product approach** — choose predictive *or* adaptive based
       on the product nature.
    2. **PM system approach** — must be adaptive *regardless* of
       product, because it manages complex human relations. Use a
       base PM system and enrich it gradually; if not, at least
       revise as the project progresses.

## When this applies

- Choosing a development life cycle (predictive / iterative /
  incremental / adaptive / hybrid)
- Defending the choice against fashion-driven pressure ("we must
  go Agile" or "Agile is just chaos")
- Designing the PM system itself: always reserve room for
  adaptation; don't lock it in at kickoff
- Hybrid projects: predictive for some components, adaptive for
  others

## Anti-patterns

- Forcing Agile on a fully predictable, design-once-build-once project
- Forcing Predictive on a high-uncertainty user-driven product
- Calling PMBOK 7 "Agile" or "obsolete because of Agile"
- Locking the PM system upfront and refusing to adapt it
