# Enable change to achieve the envisioned future state

**TL;DR:** Distinguish **output** (what the project builds) from
**outcome** (what changes in the world). Outputs only matter when
they cause adoption and outcomes; outcome management is largely
program management, but the project still has to align and report.

## Core ideas

- **Output ≠ Outcome.** Implementing a document-management system is
  the *output*; *easier and more efficient access to documents* is
  the *outcome*. Confusing the two is a common source of project
  failure.
- **Project produces output; outcomes need more.** Outcomes usually
  require additional activities (training, change-management,
  follow-on projects). Therefore outcomes are managed primarily at
  the **program management** layer.
- **Project still has responsibilities for outcomes.**
    1. Understand *why* the project exists and what outcomes it is
       meant to enable.
    2. Align project activities with that intent.
    3. Provide the program/portfolio layer with information they
       need to manage the outcomes.
- **Reframing for the PM:** see yourself as a *change enabler*, not
  a deliverable factory. Successful project = a project whose
  output produces the intended changes in stakeholder behavior and
  organizational state.

## When this applies

- Defining project success criteria (outcome-aware, not just output)
- Drafting a benefits-realization plan
- Stakeholder change-management planning around delivery
- Reporting up to program/portfolio level: include adoption signals
- Choosing whether activities like training/communication belong
  inside the project, in a program, or as a separate project

## Anti-patterns

- Project marked "successful" on output delivery while the outcome
  fails to materialize (no adoption)
- "Throw it over the wall" delivery → no follow-up, no training
- PM optimizes only for output metrics, ignores outcome signals
- Treating the project as standalone without program/portfolio context
