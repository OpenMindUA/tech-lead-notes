# Collaborative team environment

**TL;DR:** A great project needs a great *team*, not just great
individuals. Build the team through real work, not corporate rituals.

## Core ideas

- **Team-building is embedded, not separate.** Replace lone planning
  with facilitated workshops. Use any work activity as a chance to
  improve relationships and shared understanding.
- **Use methodology hooks for collaboration.** P3.express monthly
  closing has stakeholder-satisfaction evaluation + group
  improvement-planning (e.g. via Delphi). Group involvement creates
  buy-in and surfaces ideas a single PM would miss.
- **Common understanding requires explicit structure.** For projects
  past trivial size:
    - clarify what each person does and what they can expect from
      others (PRINCE2 "defined roles & responsibilities")
    - formalize work flow (who reviews what, in how many days)
    - agree explicit *ground rules* of behavior
- **Facilitate the team to set rules, do not dictate.** Top-down
  rules feel bossy and get rejected; team-set rules are realistic
  and adopted. Example: P3.express Code of Conduct was crowd-sourced.

## When this applies

- Kicking off a new project team
- Planning workshops, retrospectives, decision workshops
- Setting up review/approval flows in a methodology
- Drafting a project charter or team agreement

## Anti-patterns

- "Team-building" trips/games disconnected from real work
- PM writes rules, hands them down, expects compliance
- Treating planning as a one-person task to be efficient
- No agreed flow → constant rework and unclear handoffs
