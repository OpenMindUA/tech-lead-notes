# Prefer results and the truth to affiliations

**TL;DR:** Don't be tribal about a methodology. Use what works for the
project; learn from systems you don't even use.

## Core ideas

- **Group identity distorts choices.** Loyalty to a method
  ("I'm a Scrum person", "I'm a PRINCE2 expert") biases evaluation
  away from results.
- **Be a polyglot of practices.** Learn multiple systems; pick the
  most appropriate per project.
- **Cross-pollinate.** Even systems you don't adopt can inspire
  improvements to the system you do use.

## When this applies

- Choosing a methodology for a new project
- Resolving methodology debates in an organization
- Designing a tailored system that draws from multiple sources
  (cf. holistic tailoring, not Frankenstein cherry-picking)

## Anti-patterns

- "We're an Agile shop, we don't use anything from PRINCE2"
- Discrediting a method without reading it
- Reducing professional identity to a single framework
