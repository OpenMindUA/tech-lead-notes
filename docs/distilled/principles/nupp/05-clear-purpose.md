# Don't do anything without a clear purpose

**TL;DR:** Every practice/artifact must have a purpose you can name.
Otherwise it's cargo-cult work — looks like methodology, produces
nothing.

## Core ideas

- **Counterfactual test.** Imagine two worlds, identical except for
  this practice/artifact. Is the difference worth the effort?
  If you can't articulate the difference, skip the practice.
- **Without purpose, two failure modes:**
    1. The practice has no benefit *in your context*; you waste effort.
    2. The practice has potential benefit, but you do it
       mechanically and miss the benefit.
- **Templates are dangerous tools.** A PRINCE2 *highlight report*
  template, filled in mechanically, ≠ an effective highlight
  report. Read the methodology, understand the purpose, build a
  document that *serves* the purpose, then compare to the template
  and reconcile differences only when you understand them.

## When this applies

- Adopting any new artifact/ceremony from a methodology
- Deciding whether to keep an existing practice that nobody likes
- Building tailored templates from scratch instead of copy-pasting
- Code reviews of a methodology rollout

## Anti-patterns

- Downloading a template, filling it, sending it on schedule
- Adopting Scrum/PRINCE2/etc. ceremonies without articulating why
- Compliance theater: doing it because the standard says so
