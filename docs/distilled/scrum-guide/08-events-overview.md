# Scrum events — overview

**TL;DR:** Five events. The **Sprint** is a container for the other
four (**Sprint Planning, Daily Scrum, Sprint Review, Sprint
Retrospective**). Each event is a **formal opportunity to inspect
and adapt** an artifact. Skipping or skimping an event = lost
inspect/adapt opportunity.

## Why events exist

- They **operationalize the three pillars** (transparency,
  inspection, adaptation).
- They **create regularity** so teams don't need ad-hoc meetings.
- All events should ideally happen at **the same time and place** to
  reduce complexity.

## The five events

| # | Event | Timebox (1-month Sprint) | Purpose |
|---|-------|--------------------------|---------|
| 1 | [The Sprint](09-the-sprint.md) | up to **1 month** | Container; turns ideas into value |
| 2 | [Sprint Planning](10-sprint-planning.md) | up to **8 h** | Plan the Sprint: Why / What / How |
| 3 | [Daily Scrum](11-daily-scrum.md) | **15 min** | Inspect progress toward Sprint Goal; adapt the day's plan |
| 4 | [Sprint Review](12-sprint-review.md) | up to **4 h** | Inspect outcome with stakeholders; adjust Backlog |
| 5 | [Sprint Retrospective](13-sprint-retrospective.md) | up to **3 h** | Inspect process; plan improvements |

For shorter Sprints, all timeboxes scale down proportionally.

## When this applies

- Designing a Scrum cadence calendar
- Resolving "we don't have time for [event]" pushback
- Onboarding stakeholders to the Sprint Review

## Anti-patterns

- Cutting events to "make room" for more work
- Daily Scrum as a status meeting for the manager
- Sprint Review as one-way demo (not a working session)
- Retrospective skipped or kept superficial — kills improvement loop
- Events at random times — destroys regularity
