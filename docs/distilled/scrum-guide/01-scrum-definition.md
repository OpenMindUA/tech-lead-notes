# Scrum definition

**TL;DR:** Scrum is a **lightweight framework** for delivering value
on **complex problems** through small cross-functional teams,
**iterative Sprints**, and continuous inspection-and-adaptation.
Purposefully **incomplete** — defines just enough rules to enable
empirical work.

## Core loop (one Sprint)

The Scrum Master fosters an environment where:

1. **Product Owner** orders work into a **Product Backlog**.
2. **Scrum Team** turns a selection into a working **Increment** during
   the **Sprint**.
3. Team and stakeholders **inspect** results and **adjust** for the
   next Sprint.
4. **Repeat.**

## Properties

- **Complex-problem focus.** Scrum addresses problems where requirements
  and approach are not predictable up-front.
- **"Developers" includes everyone.** The word is used inclusively
  for any specialist doing the work — researchers, analysts,
  designers, scientists, etc.
- **Framework, not methodology.** Defines roles, events, artifacts and
  rules; intentionally leaves **techniques and processes** to the team.
- **Immutable rules.** Implementing only parts of Scrum is possible
  but the result is "not Scrum" (per the End Note). Each element
  serves a specific purpose; removing one covers up problems and
  limits the benefits.
- **Wraps existing practice.** Scrum surrounds existing techniques,
  exposing what helps and what hinders so improvements can be made.

## When this applies

- New product or initiative with complex/uncertain scope
- Existing teams that want a baseline empirical framework
- Cross-functional small teams (≤10) that own a product end-to-end

## Anti-patterns

- "ScrumBut" — "we use Scrum but we skip [X]"
- Treating Scrum as a complete methodology and following it by rote
- Adding a project-manager role on top of the three accountabilities
- Splitting one Scrum Team into product/test sub-teams
- Using Scrum on routine, well-understood, predictable work where it
  has no leverage
